# Stash — Roadmap & Design Notes

## Status

**Phase 1 (MVP) — done.** Auth (JWT + first-admin bootstrap), locations/categories, item CRUD (quantity/photo/location/category/min-quantity/expiry/warranty), unified `Item`+`Barcode` model (self-issued QR / existing UPC-EAN / Matter pairing code all as one barcode type), continuous camera scan (`/scan`) with `POST /api/items/scan` upsert (match → quantity+1, unknown → external lookup + auto-create), single label PNG download, low-stock/expiring-soon dashboard, Docker Compose + Proxmox LXC deploy, backup/restore.

**Phase 2 — done.**
- External barcode lookup (Open Food Facts → UPCItemDB fallback, cached in `ProductLookupCache`).
- Bulk label sheet PDF printing (`POST /api/labels/sheet`, `pdfkit`, A4 3-column grid). Bundles a subsetted Noto Sans KR font (`apps/api/assets/fonts/`) because pdfkit's built-in fonts have no Hangul glyphs and item names rendered garbled without it.
- Dedicated stock movement history screen (`/history`, `GET /api/movements`) — filter by item/reason, item+user joined in, deep-linkable via `?itemId=` from the item detail page's "전체 보기" link.

**Phase 3 — done.**
- Expiry/warranty push notifications — done. VAPID keys stored in `Setting` (admin generates via a button on 외부 연동, same "no .env edit/restart" pattern as garage), `PushSubscription` model, `apps/api/src/lib/push.ts` (`sendPushToUser`/`sendPushToAllUsers` — Stash sends to *all* users since inventory is shared household-wide, unlike garage's per-vehicle access lists), `apps/api/src/jobs/expiryNotifications.ts` (cron `0 8 * * *`, looks `LOOKAHEAD_DAYS=3` ahead, marks `Item.expiryNotifiedAt`/`warrantyNotifiedAt` so each due date only notifies once — `items.ts`'s `PATCH` resets those flags to `null` whenever the corresponding date field is changed, otherwise editing a past-due date to a new future one would never notify again). `PushNotificationSettings` component embedded on `/backup`. `sw.js` got `push`/`notificationclick` handlers.
- Manual theme toggle — done. `lib/theme-context.tsx` (`ThemeProvider`/`useTheme`) persists to `localStorage` (`stash_theme`) and sets `data-theme` on `<html>`; `globals.css` gained `:root[data-theme="dark"|"light"]` blocks (attribute selector out-specificities the existing `@media (prefers-color-scheme: dark)` block, so explicit choice always wins, "시스템" just removes the attribute and lets the media query drive it). A blocking inline script in `layout.tsx`'s `<head>` applies the stored theme before first paint to avoid a flash of the wrong theme. `ThemeToggle` component (라이트/다크/시스템) embedded on `/backup`. Language toggle deferred until i18n itself lands (toggling between one language isn't meaningful yet).
- PWA offline shell — done. `sw.js` precaches all top-level shell routes (`/`, `/login`, `/scan`, `/items`, `/locations`, `/categories`, `/labels`, `/history`, `/backup`, `/offline`), and its fetch handler now falls back to a dedicated `/offline` page (instead of `/`) for uncached *navigations* only — non-navigation asset requests (JS/CSS chunks) fail through normally rather than resolving to an unrelated HTML page. `OfflineBanner` (window `online`/`offline` listeners) shows a persistent notice when offline. Found and fixed a real bug in the process: `AuthProvider.fetchMe()` had no error handling, so a network failure (e.g. actually offline) left the app stuck on "불러오는 중..." forever instead of rendering the cached shell — it now caches the last-known user in `localStorage` (`stash_cached_user`) and falls back to it on fetch failure, only clearing on an explicit 401.
- ko/en i18n — done. `apps/web/lib/i18n/translations.ts` (flat key → `{ko, en}` dictionary, ~140 keys) + `lib/i18n/locale-context.tsx` (`LocaleProvider`/`useLocale`, persists to `localStorage` as `stash_locale`, `t(key, params?)` with `{param}` interpolation, `formatDateTime(iso)` using `Intl`/`toLocaleString` keyed off the *app's* selected language rather than the browser's, so switching in-app language also re-formats timestamps). `LanguageToggle` (한국어/English) embedded on `/backup` next to the theme toggle. Every page and component was migrated off hardcoded Korean strings — verified by grepping the whole `apps/web` tree for Hangul outside `lib/i18n/`, which now only turns up Korean *code comments* (intentionally left, matches the rest of the codebase's comment convention) and two data-driven exceptions that are correct to leave alone: actual user-entered content (names, item names) and `packages/shared` API error message strings (`apps/api` itself is not localized — only the `apps/web` UI layer is; a future pass could add an `Accept-Language`-aware error layer if needed). `apiJson()`'s fallback error message reads `stash_locale` directly from `localStorage` since it's a plain lib function outside any component tree.
- Push notification body text localization — done (follow-up after the initial i18n pass). `apps/api/src/lib/push.ts` gained `sendLocalizedPushToUser`/`sendLocalizedPushToAllUsers`, which take a `(locale: string) => PushPayload` builder and apply it per-subscription using each `PushSubscription.locale` (captured at subscribe time) rather than one fixed payload for everyone — additive, the original `sendPushToUser`/`sendPushToAllUsers` are untouched so `routes/push.ts`'s `/test` route didn't need changes. `expiryNotifications.ts`'s `formatDday` now takes a `locale` param (defaults to `"ko"`) and a small `MESSAGES.ko`/`MESSAGES.en` dict supplies the title/body templates. Verified end-to-end against the real Docker stack: created two real `PushSubscription` rows (one `ko`, one `en`, different users) and ran the job directly — logs confirmed both were processed independently through `sendLocalizedPushToAllUsers` → `sendLocalizedPushToUser`, with payload-building succeeding for both before failing at the (expected, since the test rows used fake encryption keys) `web-push` encryption step.

**Inventory event webhook — done.** Generic outbound webhook for webhook-capable printer/label-device automations (e.g. Home Assistant). Not named after any specific product in code or UI — see below.

## Bug/behavior verification pass (done)

Found and fixed while exercising every route against the real Docker+Caddy stack (not just the happy path):

- **Duplicate barcode value** attach/create crashed with a raw Prisma 500 instead of a clean error. Also surfaced a schema bug: `Barcode` uniqueness was scoped `[itemId, value]`, but `/scan`'s lookup (`findFirst({ where: { value } })`) assumes values are globally unique — two items could otherwise silently share a barcode with ambiguous scan matching. Changed to `@@unique([value])` (migration `20260711114634_barcode_value_globally_unique`), added `isUniqueConstraintError()` (`apps/api/src/lib/prismaErrors.ts`) used by both the manual-attach route (checked before insert, so a conflict never leaves an orphaned item behind) and `/scan`'s create path (self-heals a same-barcode race into a quantity bump instead of erroring).
- **Corrupted/invalid image upload** to `/api/attachments` crashed with a raw 500 (sharp's error message leaking to the client) instead of a 400. Now caught and returns a clean validation error.
- **Attachment files were never deleted from disk** — `DELETE /api/attachments/:id` only removed the DB row, and deleting an item (which cascades attachments at the DB level) didn't touch their files either, leaking the uploads volume forever. Added `deleteUploadedFile()` (`apps/api/src/lib/uploads.ts`), used by both paths.
- **Several frontend mutation handlers had no try/catch** (`addManualBarcode`, `addMatterBarcode`, `adjustQuantity`, `updateField`, `generateQr`, `removeBarcode`, `handleDelete` on the item page; location/category create; user delete; all four settings-page handlers) — a failed request just silently did nothing with no toast, only a console error. All now catch and surface `err.message`.
- **Error toasts showed literal escaped quotes** (`"메시지"` instead of `메시지`) for any string `error` field, because `apiJson()` always `JSON.stringify`'d `body.error` regardless of whether it was already a string (only Zod validation failures are objects). Fixed to pass strings through as-is.

Verified via the actual `docker compose` + Caddy stack (not just `npm run dev`): categories CRUD, role-based 403 on admin routes, quantity floor at 0, location deletion `SetNull`s (doesn't delete) attached items, and a full backup export → restore round-trip.

## Inventory event webhook (implemented)

Admin sets one URL (`INVENTORY_WEBHOOK_URL`, Settings → 외부 연동 → "재고 이벤트 웹훅"; same `Setting` table pattern as `UPCITEMDB_API_KEY`, excluded from backup). Stash POSTs a JSON payload to it in two cases, built by `buildWebhookPayload()` in `apps/api/src/lib/webhook.ts`:

- `item.updated` — fired automatically whenever an item is created, edited, quantity-adjusted, or matched/created via `/scan` (`apps/api/src/routes/items.ts`). Fire-and-forget, not awaited, never blocks the request.
- `item.print_requested` — explicit "프린터로 출력" action (`POST /api/items/:id/print-request`), available on the item detail page and as a bulk action on `/labels`. Returns 400 with a clear error if no webhook is configured or the item has no barcode.

Payload shape: `{ event, itemId, name, quantity, unit, locationId, locationName, barcodeValue, symbology, labelImageUrl, timestamp }`. `barcodeValue`/`symbology` alone are enough for the receiving automation to render its own barcode — `labelImageUrl` (pointing at `GET /api/barcodes/:id/label.png`) is sent as a convenience only, not required. That endpoint was made **intentionally unauthenticated** (moved to `publicBarcodeRoutes` in `barcodes.ts`, registered separately in `index.ts`) so external automations can fetch it without a token; barcode ids are unguessable cuids, so this was treated as an auth-relaxation decision requiring explicit user sign-off (given) rather than something to decide unilaterally — worth remembering if this endpoint is touched again.

UI copy is deliberately generic ("재고 이벤트 웹훅", "웹훅을 받을 수 있는 프린터·라벨 기기 자동화") rather than naming specific integrations, since any webhook-triggered automation can consume this — not just the two below. The settings page has a "전송되는 데이터 보기" `<details>` block listing every payload field for whoever is building the receiving automation.

**Context on why this exists** (kept for reference, not required reading to use the feature): the repo owner's own Home Assistant custom components `hass-niimbot` (BLE thermal printer, one-shot print) and `hass-gicisky` (BLE e-paper shelf label, persistent — needs refreshing on every data change, which is why `item.updated` fires on every change rather than only on an explicit action) both expose their print service as HA's declarative **imagespec** element format over `POST /api/services/<domain>/<service>`. Stash deliberately doesn't speak imagespec or know about `device_id`/HA tokens — that's entirely the receiving automation's job, filtering incoming webhook events by `itemId` per physical device if needed.

Tests: `apps/api/src/lib/webhook.test.ts` covers `buildWebhookPayload`/`buildLabelImageUrl` (primary-barcode selection, no-barcode case, URL construction) without hitting the network or DB. `apps/api/src/lib/barcodeSymbology.test.ts` covers `guessSymbology`. Going forward, add a `*.test.ts` next to new pure/extractable logic as it's written — established as the ongoing convention for this project, not a one-off. `apps/web` has no test setup yet (would need `@testing-library/react` + jsdom); not started.
